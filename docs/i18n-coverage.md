## Reading the columns

- **Translated** — keys whose value is **different** from the EN message. This is the true coverage signal.
- **Identical-to-EN** — keys that ship byte-identical to EN. Some are intentional (brand names: *Astra Deck*, *Astra Downloader*, *DeArrow*, *SponsorBlock*, *YouTube*; technical formats: *MP4*, *M4A*, *VP9*, *AV1*, *H.264*, *TrustedTypes*, *ETA*). Others may be untranslated placeholders awaiting native-speaker review.
- **Missing** — keys absent from the locale file. `chrome.i18n.getMessage` will fall through to EN for these.
- **Coverage** — same as the Translated percentage; restated as the single headline number.

## How to improve coverage

1. Open `scripts/generate-locales.js` and find the locale's `T['<locale>']` translation table.
2. Add an entry mapping the English message to the translated message. Brand names + technical formats should NOT be translated.
3. Re-run `node scripts/generate-locales.js` to rewrite `extension/_locales/<locale>/messages.json`.
4. Re-run `node scripts/i18n-coverage.js` to refresh this report.
5. For zh_CN, the locale is hand-maintained (out of the generator); edit `extension/_locales/zh_CN/messages.json` directly.
